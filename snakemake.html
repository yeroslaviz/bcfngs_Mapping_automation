snakemake

Author

AY

Published

March 19, 2026

This part is still work in progress.

This repository uses snakemake for:

  1. Genome download + indexing (STAR, bwa, bowtie2) using versioned index directories, stable symlinks, and per-organism index-manifest.json reports.
  2. Mapping (currently STAR) using per-project run settings.

For non-experts, prefer the single wrapper command ./scripts/bcfngs (runs checks, then snakemake).

By default, ./scripts/bcfngs uses the micromamba environment at ~/miniconda3/envs/update_genomes. If your environment is elsewhere, add --env-prefix /path/to/env.

Quickstart

# 1) create/update env (only once, then as needed)
./scripts/bcfngs env create
./scripts/bcfngs env check

# 2) (optional) check if a newer Ensembl release exists
./scripts/bcfngs genomes check-updates

# 3) preflight checks + snakemake lint + dry-run
./scripts/bcfngs genomes plan

# 4) build genomes + indices (real run)
./scripts/bcfngs genomes build --cores 20

Configuration files

The workflow configuration is split into two files:

  1. Genome and indexing configuration: configs/genomes.yaml
    • Organism identifiers are scientific names (e.g. Homo_sapiens, Mus_musculus).
    • Genome sources are pinned to a specific Ensembl release (at the moment: 115).
    • Index output directories are versioned via index_versions. These versions are called via the bcfngs script and are not hard-coded here anymore.
  2. Per-project run configuration (mapping): configs/run.yaml
    • Project ID, input path, organism, threads, etc.

How-to (cookbook)

Build all genomes + indices

./scripts/bcfngs genomes plan
./scripts/bcfngs genomes build --cores 20

Build only specific genomes

./scripts/bcfngs genomes build --cores 20 --organism Homo_sapiens,Mus_musculus

Ensembl has a newer release

  1. Check:
./scripts/bcfngs genomes check-updates
  1. If an update is available, edit configs/genomes.yaml and bump:
  • releases.ensembl: 116 → 117 (etc.)
  1. Then:
./scripts/bcfngs genomes plan
./scripts/bcfngs genomes build --cores 20

New STAR/bwa/bowtie2 version (rebuild indices)

  1. Update tools:
./scripts/bcfngs env update
./scripts/bcfngs env check
  1. Rebuild with the actual tool version or pin the version label explicitly:
  2. Update configs/genomes.yaml:
  • set index_versions.star, index_versions.bwa, index_versions.bowtie2 (these strings are used in output folder names)

Index version are now recognized automatically via the bcfngs script.

  1. Rebuild:
./scripts/bcfngs genomes build --cores 20

Examples:

./scripts/bcfngs genomes build --cores 20 --star-index-version 2.7.11b
./scripts/bcfngs genomes build --cores 20 --star-index-version 2.7.10b --organism Homo_sapiens

The wrapper now requires explicit version labels to match the resolved binary version for that tool.

Use HPC modules for aligners (fallback)

This still uses the micromamba env for python, snakemake, and curl, but resolves aligners from modules:

module load bclconvert/4.3.13 star bowtie2 bwa
./scripts/bcfngs --profile modules env check
./scripts/bcfngs --profile modules genomes plan
./scripts/bcfngs --profile modules genomes build --cores 20

If you preload an older module version, the wrapper reuses it and prints the resolved tool/version report to stderr. Mapping then requires the matching explicit index version flag, for example --star-index-version 2.7.10b.

Genome download + indexing

Workflow file: workflow/genomes.smk

Mapping the samples with the STAR

mapping the samples in a paired-end format

This script contains three (four) steps. in the first step the fastq samples are mapped to the indexed genome. Then, they are being indexed. Last, after creating a chromosome-size file, the bam files are converted into bigwig files for better visualization using the UCSC browser.

We have two separate scripts for single-end and paired-end runs. The PE run looks for both R1 and R2 fastq files, while the SE script will ignore all R2 samples.

After the environment was created and activated, one must change to the directory of the concatenated fast files. E.g.

cd /fs/pool/pool-bcfngs/fastq_files/P000/P000_Testrun/conc.fastq/

To run mapping, update configs/run.yaml in the repo, then run one of these Snakefiles.

# for single-end samples
snakemake -nps /path/to/repo/Star.MappingQuant.SingleEndFastq.Snakefile -j 50
# for paired-end samples
snakemake -nps /path/to/repo/Star.MappingQuant.PairedEndFastq.Snakefile -j 50
  • -n - dry-run. This parameter will only output the command, but execute them. This should be used to test the parameters, but must be removed for the analysis to be done.
  • -p - Print the commands to be execute.
  • -s - the name of the snakeake file to be executed.
  • -j - #cores. How many cores should be used in the run.
