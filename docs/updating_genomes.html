Updating genomes and indices

Author

AY

Published

March 6, 2026

This page is the operator guide for ./scripts/bcfngs genomes. It covers:

  • checking whether the pinned genome release is outdated
  • planning and building genome sources plus indices
  • rebuilding only selected organisms
  • updating genome configuration and versioned index directories
  • advanced direct snakemake usage when the wrapper does not expose a knob

Environment setup is documented elsewhere. This page assumes ./scripts/bcfngs already runs on your machine or cluster.

1 Command overview

bcfngs genomes currently exposes three subcommands:

  • check-updates: compare the pinned Ensembl release in configs/genomes.yaml with the latest Ensembl release visible on ftp.ensembl.org
  • plan: run preflight checks, snakemake --lint, and a dry-run
  • build: run plan, then execute the workflow for real

Inspect the wrapper help first. Current output from the script:

./scripts/bcfngs genomes --help
usage: bcfngs genomes [-h] {check-updates,plan,build} ...

positional arguments:
  {check-updates,plan,build}

optional arguments:
  -h, --help            show this help message and exit
./scripts/bcfngs genomes plan --help
usage: bcfngs genomes plan [-h]

optional arguments:
  -h, --help  show this help message and exit
./scripts/bcfngs genomes build --help
usage: bcfngs genomes build [-h] [--cores CORES] [--organism ORGANISM]

optional arguments:
  -h, --help           show this help message and exit
  --cores CORES
  --organism ORGANISM  Comma-separated subset of organism keys (e.g.
                       Homo_sapiens,Mus_musculus)
./scripts/bcfngs genomes check-updates --help
usage: bcfngs genomes check-updates [-h]

optional arguments:
  -h, --help  show this help message and exit

That means:

  • use --cores to control total workflow parallelism
  • use --organism for a comma-separated subset of organism keys
  • use direct snakemake for advanced cases such as scheduler memory flags or building only one index type

2 Shared wrapper parameters

These apply to all three genomes subcommands and are cross-referenced from the chapters below.

Parameter Current default Applies to Notes
--profile micromamba check-updates, plan, build modules prepends module load bclconvert/4.3.13 star bowtie2 bwa, but still runs python and snakemake through micromamba.
--env-prefix ./.mamba/env check-updates, plan, build This is the current default in scripts/bcfngs. Override it if your environment lives elsewhere.
configs/genomes.yaml required check-updates, plan, build Release pins, organisms, output root, thread defaults, RAM defaults, and index version strings all come from this file.

3 Common config fields

These are the main configs/genomes.yaml fields referenced by the three command chapters.

Config field Current value Used by Notes
genomes_root /fs/pool/pool-bcfngs/genomes plan, build plan checks write access here. build writes sources, indices, and symlinks here.
releases.ensembl 116 check-updates, plan, build Main Ensembl release pin. check-updates compares this to ftp.ensembl.org.
releases.ensemblgenomes.metazoa 61 plan, build Used for ensemblgenomes organisms such as Drosophila_pseudoobscura.
default_threads 16 build Used by STAR genome generation and bowtie2-build.
default_ram_bytes 168632718037 build Passed to STAR --limitGenomeGenerateRAM.
index_versions.star "unknown" build Controls versioned output paths such as starIndex-<version>.
index_versions.bwa "unknown" build Controls versioned output paths such as bwaIndex-<version>.
index_versions.bowtie2 "unknown" build Controls versioned output paths such as bowtie2Index-<version>.
organisms.* all configured organisms plan, build Scientific-name keys such as Homo_sapiens, Mus_musculus, Drosophila_melanogaster.
organisms.*.sa_index organism-specific build Passed to STAR --genomeSAindexNbases.
organisms.*.fasta_url / gtf_url optional explicit URLs plan, build Used for organisms that do not fit the generated Ensembl filename pattern.

4 check-updates

4.1 Parameters and defaults

Item Current default / value Where set Notes
Shared wrapper parameters see Shared wrapper parameters CLI --profile and --env-prefix still apply here.
Command-specific flags none CLI This subcommand has no extra runtime flags beyond -h/--help.
Compared release pin releases.ensembl: 116 configs/genomes.yaml This is the pinned release compared to the latest Ensembl release found on https://ftp.ensembl.org/pub/.
Network target https://ftp.ensembl.org/pub/ internal behavior If it is unreachable, the command reports the current pin but cannot confirm the latest release.
Writes to disk none internal behavior Report only. No files are edited.
Not checked automatically releases.ensemblgenomes.*, explicit fasta_url, explicit gtf_url config Review these manually when working with EnsemblGenomes or custom URLs.

4.2 Micromamba / conda-style environment

Use this when the wrapper should run in its default micromamba profile:

./scripts/bcfngs genomes check-updates

Typical output looks like:

Ensembl latest release: 117  current pinned: 116  status: UPDATE_AVAILABLE
Suggested bump: set releases.ensembl to 117 in configs/genomes.yaml

4.3 Modules profile

If you want the same command to run under the modules profile:

module load bclconvert/4.3.13 star bowtie2 bwa
./scripts/bcfngs --profile modules genomes check-updates

For check-updates, the modules profile usually does not change the result. It is only relevant if you are keeping your operational workflow consistent across all genomes commands.

4.4 If an update is available

  1. Edit configs/genomes.yaml.
  2. Bump releases.ensembl, for example 116 to 117.
  3. Review any organism entries that use explicit fasta_url or gtf_url.
  4. Continue with plan and then build.

If you are updating an EnsemblGenomes organism such as Drosophila_pseudoobscura, also review releases.ensemblgenomes plus the explicit URLs in its organism block.

5 plan

5.1 Parameters and defaults

Item Current default / value Where set Notes
Shared wrapper parameters see Shared wrapper parameters CLI --profile and --env-prefix apply here.
Command-specific flags none CLI The wrapper exposes no extra plan flags.
Organism scope all configured organisms wrapper behavior The wrapper plans the full configured set. Use direct snakemake with --config org_subset=... if you want a subset dry-run.
genomes_root /fs/pool/pool-bcfngs/genomes configs/genomes.yaml plan checks that this location is writable.
Release pins and URLs see Common config fields config plan resolves all FASTA/GTF URLs from the current release pins and organism blocks, then tests them with curl -fsSI.
Underlying Snakemake steps snakemake --lint then snakemake -npr wrapper behavior plan is both a lint pass and a dry-run.

5.2 Local run with micromamba / conda-style environment

Use this before every real build:

./scripts/bcfngs genomes plan

This checks that:

  • configs/genomes.yaml parses
  • genomes_root is writable
  • each configured FASTA/GTF URL responds
  • workflow/genomes.smk passes snakemake --lint
  • the workflow dry-run resolves successfully

5.3 Local run with modules

If your cluster provides aligners via modules:

module load bclconvert/4.3.13 star bowtie2 bwa
./scripts/bcfngs --profile modules genomes plan

The modules profile still uses the micromamba environment for python and snakemake; it only changes how tool discovery happens for the aligners.

5.4 Direct Snakemake subset dry-run

The wrapper does not expose a --organism option for plan. Keep the direct snakemake form when you need a subset dry-run:

snakemake -npr \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml \
  --config org_subset=Homo_sapiens,Mus_musculus

This is useful for:

  • testing a release bump on one organism first
  • checking a new organism block before rebuilding everything
  • isolating failures in one species without noise from the rest of the config

5.5 SLURM-oriented dry-run example

For clusters where you want to inspect scheduler-facing settings before a real run, keep the direct snakemake example:

snakemake -npr \
  --executor slurm \
  --jobs 20 \
  --default-resources mem_mb=180000 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml

plan itself does not submit jobs in dry-run mode, but this is still useful to check that your SLURM executor configuration parses correctly.

6 build

Item Current default / value Where set Notes
Shared wrapper parameters see Shared wrapper parameters CLI --profile and --env-prefix apply here too.

--organism all configured organisms CLI Optional comma-separated subset such as Homo_sapiens,Mus_musculus.

default_ram_bytes 168632718037 configs/genomes.yaml Passed to STAR --limitGenomeGenerateRAM.

6.2 Local run with micromamba / conda-style environment

Build all configured genomes:

./scripts/bcfngs genomes build --cores 20

More local examples:

./scripts/bcfngs genomes build --cores 8
./scripts/bcfngs genomes build --cores 20
./scripts/bcfngs genomes build --cores 32

--cores is the total Snakemake job budget. It is not the same thing as default_threads, which controls per-tool threading inside rules such as STAR genome generation.

6.3 Local run with modules

module load bclconvert/4.3.13 star bowtie2 bwa
./scripts/bcfngs --profile modules genomes build --cores 20

This is the best fit when the aligners should come from modules but python and snakemake should still come from the micromamba environment.

6.4 Build only selected organisms

Organism names must match the keys in configs/genomes.yaml exactly.

./scripts/bcfngs genomes build --cores 20 --organism Homo_sapiens
./scripts/bcfngs genomes build --cores 20 --organism Homo_sapiens,Mus_musculus
./scripts/bcfngs genomes build --cores 12 --organism Drosophila_melanogaster,Drosophila_pseudoobscura

Use this when:

  • you added or changed only one organism
  • you want to test a release bump on a smaller subset first
  • you do not want to rebuild every organism after a config edit

6.5 SLURM-oriented build example

The wrapper does not expose --executor, --mem, or --mem-per-cpu. Keep the direct snakemake form for cluster-scheduled runs:

snakemake -pr \
  --executor slurm \
  --jobs 20 \
  --default-resources mem_mb=180000 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml

If your site schedules by memory per CPU instead:

snakemake -pr \
  --executor slurm \
  --jobs 20 \
  --default-resources mem_mb_per_cpu=8000 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml

Adjust these examples to your local cluster conventions. The current ./scripts/bcfngs genomes build command does not parse --mem or --mem-per-cpu directly.

6.6 Memory and threading

There are two different layers to tune:

  • --cores: total workflow parallelism for the outer Snakemake run
  • default_threads: per-rule thread count used inside STAR genome generation and bowtie2-build

Typical patterns:

  • 20-core workstation, one large STAR job at a time: set default_threads: 20 and run ./scripts/bcfngs genomes build --cores 20
  • 20-core workstation, allow two medium jobs in parallel: set default_threads: 10 and run ./scripts/bcfngs genomes build --cores 20
  • larger-memory node for STAR genome generation: increase default_ram_bytes

For a permanent STAR memory change, edit:

default_ram_bytes: 250000000000

6.7 Build only one index type with direct Snakemake

The wrapper always targets the full workflow. To build only one tool, target the desired output file directly.

Replace each version placeholder with the matching value from index_versions in configs/genomes.yaml.

Only STAR for Homo_sapiens:

snakemake -pr -j 20 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml \
  /fs/pool/pool-bcfngs/genomes/Homo_sapiens/starIndex-<STAR_VERSION>/SA

Only bwa for Homo_sapiens:

snakemake -pr -j 20 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml \
  /fs/pool/pool-bcfngs/genomes/Homo_sapiens/bwaIndex-<BWA_VERSION>/Homo_sapiens.bwt

Only bowtie2 for Homo_sapiens:

snakemake -pr -j 20 \
  -s workflow/genomes.smk \
  --configfile configs/genomes.yaml \
  /fs/pool/pool-bcfngs/genomes/Homo_sapiens/bowtie2Index-<BOWTIE2_VERSION>/Homo_sapiens.1.bt2

These commands still pull in prerequisite downloads, but they avoid rebuilding every index type.

6.8 Rebuild after a genome release change

After check-updates reports a newer Ensembl release:

  1. Edit configs/genomes.yaml.
  2. Bump releases.ensembl, for example 116 to 117.
  3. Review any explicit fasta_url or gtf_url entries.
  4. Run plan.
  5. Run build.

If you want a cautious rollout, start with one organism:

./scripts/bcfngs genomes build --cores 20 --organism Homo_sapiens

6.9 Rebuild after a tool version change

When the genome sequence did not change but an indexer changed, update only the relevant version labels:

index_versions:
  star: "2.7.11b"
  bwa: "0.7.18"
  bowtie2: "2.5.5"

Then rerun:

./scripts/bcfngs genomes plan
./scripts/bcfngs genomes build --cores 20

Successful builds update the stable symlinks automatically:

  • /fs/pool/pool-bcfngs/genomes/<Organism>/starIndex
  • /fs/pool/pool-bcfngs/genomes/<Organism>/bwaIndex
  • /fs/pool/pool-bcfngs/genomes/<Organism>/bowtie2Index

These stable names point to versioned directories such as starIndex-2.7.11b.
